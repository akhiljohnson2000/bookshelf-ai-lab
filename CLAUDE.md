# BookShelf
A personal book-catalogue REST API (a simplified Goodreads) with a React frontend.
Browse and search books; shelves and reviews exist as seed data but have no
endpoints yet. Monorepo; all data lives in JSON files ‚Äî there is no database.

# Commands
- `npm run dev` ‚Äî API on `http://localhost:4000` (base path `/api`)
- `npm run dev:web` ‚Äî Vite/React frontend on `http://localhost:5173` (calls the API via CORS)
- `npm run build` ‚Äî build shared package, then API ¬∑ `npm run lint` ‚Äî ESLint
- `npm test` ‚Äî Vitest unit tests (`apps/api`), run against mocked data access

# Tech Stack
- **Runtime:** Node.js 20+ (ESM ‚Äî `"type": "module"`)
- **API:** Express 4 + TypeScript, run in dev with `tsx watch`
- **Web:** Vite + React 18 + Tailwind (`apps/web`)
- **Data:** JSON files in `/data` (`books.json`, `shelves.json`, `reviews.json`) ‚Äî no DB
- **Workspaces:** npm workspaces (`apps/*`, `packages/*`)

# Architecture
Thin routes delegate to services; services use a JSON data-access layer. Shared
types and error classes live in a separate package consumed by the API.
- `apps/api/src/routes` ‚Äî Express route handlers (thin; parse input, call services)
- `apps/api/src/services` ‚Äî business logic (filtering, pagination, search)
- `apps/api/src/data` ‚Äî JSON data-access layer (`books.repository.ts`, `jsonStore.ts`)
- `apps/api/src/middleware` ‚Äî validation, request logging, error handling
- `apps/api/src/config` ‚Äî env/config + resolved data-file paths
- `packages/shared` ‚Äî shared types + `AppError`/`ValidationError`/`NotFoundError`/`ConflictError`

# Conventions
- **Response types live in `packages/shared` ‚Äî import them, don't hand-roll shapes:**
  - `SuccessResponse<T>` ‚Üí `{ success: true, data }` (single resource; `201` on create)
  - `PaginatedResponse<T>` ‚Üí `{ success: true, data, meta }` (list endpoints, e.g. `GET /api/books`)
  - `ErrorResponse` ‚Üí `{ success: false, error: { message, code?, details? } }`
- **Errors:** `throw` an `AppError` subclass; the central `errorHandler` formats the
  response. Never `res.send` raw strings or build error JSON by hand in a route.
- **Validation** lives in middleware (e.g. `validateBookPayload`), not route handlers.
  `ValidationError`'s `details` carries field-level messages, e.g. `{ title: "title is required" }`.
- Early returns over nested `if/else`; named exports over default exports.
- No `any` in TypeScript ‚Äî use `unknown` and narrow.
- **IDs** are prefixed, zero-padded strings: `book_001`, `shelf_001`, `review_001`.
- **File naming:** entity-scoped modules use `entity.role.ts` (`books.repository.ts`,
  `books.service.ts`, `books.routes.ts`); generic utilities use camelCase
  (`jsonStore.ts`, `errorHandler.ts`). This split is intentional ‚Äî don't "fix" it.

# Data Layer
- All reads/writes go through `apps/api/src/data/` ‚Äî never touch `fs` from routes or services.
- Use `readJson`/`updateJson` from `jsonStore.ts`; writes are atomic (temp file + `rename`).
- New `book_NNN` IDs are generated by incrementing the highest existing id (see `books.repository.ts`).
- Mutations that must reject duplicates throw `ConflictError` from inside the `updateJson` updater.

# Testing
- Vitest (`npm test`), AAA pattern (Arrange/Act/Assert). Test files are excluded from the build.
- Mock the data-access layer (`vi.mock('../data/...')`) ‚Äî don't read real JSON files in tests.
- Place test files next to source: `books.service.ts` ‚Üí `books.service.test.ts`.

# What NOT to do
- Don't add a real database ‚Äî the JSON file store is deliberate for this project.
- Don't add new npm dependencies without discussing first.
- Don't read or write `/data/*.json` directly from routes or services ‚Äî use the data layer.
- Don't use `console.log` ‚Äî use the `requestLogger` middleware, or remove debug output.
- Don't modify `packages/shared` without checking consumers in `apps/api`.
